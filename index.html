webtoon · AI 웹툰 자동화 파이프라인 (k8s ComfyUI 풀 + Ray Serve VLM)

스토리보드 → 렌더 → 비전 critic → reroll/repair → 등록까지 각 단계를 JSON 산출물로 분리한 AI 웹툰 제작 파이프라인입니다. 로컬 Mac은 오케스트레이터 역할만 하고, 모든 GPU 연산은 Keiailab 사내 Kubernetes 클러스터(ml 네임스페이스)로 fan-out합니다. seed/prompt/모델 버전을 기록해 재현성을 확보하고, 컷 계약과 캐릭터 identity를 추적해 일관성을 유지합니다. Higgs Field 기반 이미지·동영상 생성과 FLUX 계열 이미지 생성/수정 실험을 함께 사용해 컷 후보, 영상화 후보, 보정 후보를 자동 생성하는 흐름도 구성했습니다.

① GPU 호출 (렌더): ComfyUI를 풀 2종으로 운영합니다. comfyui-sdxl ×2(RTX 4070 12GB)는 SDXL(Illustrious / NoobAI / Animagine) + ControlNet(pose/depth/canny) 스테이지를, comfyui-large ×2(RTX 4060 Ti 16GB)는 Qwen-Image GGUF / Z-Image 등 VRAM이 큰 잡을 담당합니다. 클라이언트는 MURIM_COMFY_SDXL_ENDPOINTS / MURIM_COMFY_LARGE_ENDPOINTS로 pod별 엔드포인트 풀을 잡아 least-busy 선택 + 컷 단위 병렬로 호출합니다(단일 GPU 직렬 대비 wall-clock 약 1/4, 16GB 풀로 OOM 해소). 실제 렌더 그래프는 Qwen-Image-Edit 2511 fp8 + Lightning LoRA 기반으로, 확산 모델 / CLIP / VAE 로드 → TextEncodeQwenImageEditPlus → KSampler → VAE 디코드로 구성됩니다. 컷별 이미지 후보는 FLUX/ComfyUI 경로와 Higgs Field 생성 결과를 함께 비교하고, 영상 후보는 Higgs Field 기반 생성 결과를 검수 루프에 연결했습니다.

② LLM / VLM 호출: 같은 클러스터의 KubeRay + Ray Serve(OpenAI 호환, :8000/v1)에서 비전·텍스트 모델을 서빙합니다. qwen3-vl-30b-a3b(MoE)를 critic 주력 + 스토리보드 writer + reroll 프롬프트 수선에, internvl3-5-8b를 hero 컷 2차 교차검증에, qwen3-4b를 리포트/등록 payload에 사용합니다. STORY_MODEL=k8s로 라우팅하고 외부 codex CLI는 fallback으로 둡니다. critic은 에피소드 단위로 asyncio 동시 호출해 15컷 직렬 2–4min을 30s 미만으로 단축합니다.

③ 서버 인프라 / GPU 스케줄링: NVIDIA GPU Operator + HAMi DRA driver로 ComfyUI/Ray 워커가 총 16장 GPU 노드 풀(4060 Ti / 4070 / 4070 Ti)에서 GPU를 DRA claim 방식으로 공유하고, 유휴 워커는 scale-to-zero 됩니다. 모델 자산은 공유 /models/comfyui(Ceph) 볼륨에 두고 ml-model-download-* Job으로 체크포인트·ControlNet·LoRA를 주입하며, 클라이언트는 MURIM_MODEL_PROFILE=k8s 계약으로 동일 경로를 가정합니다. Mac↔클러스터는 VPN(utun100)으로 내부 API(10.31.10.10:6443)·ClusterIP(10.43/16)에 접속하고, 엣지는 Cilium Gateway가 담당합니다.

④ 파이프라인 흐름: 시놉시스 → 스토리보드(VLM) → renderability gate(결정적) → ComfyUI 풀 fan-out 렌더 → VLM critic(병렬, hero는 교차검증) → action router(결정적) → reroll 표적 수선(실패 이미지 입력) → Qwen image-edit repair → 자동 re-critic·등록·QA 리포트 → operator 최종 승인 → chocomics 플랫폼 발행. 사람은 시놉시스 작성·repair 승인·최종 승인만 맡고, 픽셀 개선은 자동 루프가 먼저 수행합니다.

callLLM
FastAPI · LLM benchmark tabs

서버 PC 사양에 맞춰 qwen3-8b-int4와 gemma-4-e4b 모델을 설치하고, 동일 프롬프트 기준 응답 품질과 속도를 비교 테스트한 LLM 벤치마크 도구입니다.

테스트 환경은 AMD Ryzen 5 5600G, 12 threads, RAM 31Gi, NVIDIA GeForce RTX 4060 Ti 16GB, NVIDIA Driver 580.126.09 기준으로 구성했습니다.

BIOS에서 RAM 사용 제한을 적용한 상태로 운영했고, Kubernetes GPU Operator와 NVIDIA device plugin/container toolkit을 설치해 GPU 워크로드를 실행했습니다.

FastAPI 백엔드에서 프롬프트 실행, 응답 수집, 결과 조회 흐름을 분리하고 LangChain/RAG 프로필을 연결해 모델별 실행 과정을 비교했습니다.

화면은 Browse/Workflow/Response/Results/Explain 탭으로 구성해 모델 등록, 실행 흐름, 응답 결과, 비교 이력을 한 번에 확인할 수 있게 만들었습니다.

Stitch export 결과와 샘플 프롬프트를 함께 관리해 AI 디자인 산출물을 재사용하고 비교하기 쉬운 구조로 정리했습니다.

디자인 방향성, 상호작용 효과, 레퍼런스 분석 내용을 화면 단위로 나누어 프롬프트 작성과 결과 검토 흐름이 이어지도록 구현했습니다.

n8n-project
n8n · video automation browser

n8n 자동화 흐름과 콘텐츠 피드를 웹에서 확인하는 영상/콘텐츠 자동화 프로젝트입니다.

워크플로우 탭에서 자동화 단계와 실행 결과를 확인하고, 생성된 결과물을 피드 형태로 검토할 수 있게 구성했습니다.

반복 작업은 n8n 워크플로우로 분리하고, 웹 화면은 입력, 실행 결과, 콘텐츠 목록을 확인하는 역할로 나눴습니다.

자동화 파이프라인으로 생성한 영상은 실제 유튜브 채널에 쇼츠 약 20개, 롱폼 3개 수준으로 업로드해 결과를 검증했습니다.

AI 도구와 자동화 파이프라인을 연결해 콘텐츠 수집, 처리, 결과 검토, 업로드까지 이어지는 시간을 줄이는 방향으로 구현했습니다.

keiailab.com
Next.js · 회사 공식 사이트 · k8s 플랫폼 쇼케이스

Keiailab 공식 웹사이트입니다. 일러스트 히어로 + light-ripple 캔버스 인터랙션 위에 메가메뉴(Open Source / Infra / Services / Company)를 얹은 원페이지 + 서브페이지 구조로 구현했습니다.

Open Source 메뉴에는 직접 개발·운영 중인 postgres-operator, valkey-operator, mongodb-operator와 webEmbedding(AI Design)을 정리해 제품 라인업을 한눈에 보여줍니다.

Infra 페이지(KeiInfra)는 운영 중인 프로덕션 k8s 클러스터를 kubectl에서 실측해 Ready 노드 수, Ceph Raw 용량, Pod 수와 Cilium Gateway · FluxCD · NVIDIA GPU Operator · HAMi DRA · KubeRay · ComfyUI 등 라이브 아키텍처 구성요소를 그대로 시각화합니다.

인증 인프라는 Keycloak 기반 sso.keiailab.com 경로를 NAS Keycloak 프록시로 수렴시키고, Apple 로그인에서 필요한 response_mode=form_post callback을 Kubernetes bridge 서비스로 처리하도록 구성했습니다.

각 오퍼레이터 상세 페이지는 라이선스/언어/지원 버전 배지, README, 기능, 선언형 클러스터 정의까지 문서화해 오픈소스 쇼케이스와 세일즈 랜딩을 겸하도록 설계했습니다.

quant_trading_app

한국/미국 시장을 함께 다루는 하이브리드 퀀트 자동매매 앱입니다. 5영업일 리밸런스와 문서형 ML 레이어를 중심으로 전략을 운영합니다.

백엔드는 FastAPI + SQLite(SQLAlchemy)로 시세 수집·지표 계산·백테스트·포트폴리오/지갑 영속화·봇 실행·알림 평가를 담당하고, 체결 실행은 Rust execution engine, 메시징은 Redis Streams + DLQ 재처리로 구성했습니다.

실시간 봇이 시세를 받아 전략 신호를 생성하고, 운영 화면은 "자동매매 운영 문서"와 "매매 동향 문서" 형태로 동작 프로세스·설계 의도·시장별 운영 상태·체결 흐름·일별 수익률을 문서처럼 확인하도록 만들었습니다.
