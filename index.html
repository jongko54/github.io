webtoon · AI 웹툰 자동화 파이프라인 (k8s ComfyUI 풀 + Ray Serve VLM)

스토리보드 → 렌더 → 비전 critic → reroll/repair → 등록까지 각 단계를 JSON 산출물로 분리한 AI 웹툰 제작 파이프라인입니다. 로컬 Mac은 오케스트레이터 역할만 하고, 모든 GPU 연산은 Keiailab 사내 Kubernetes 클러스터(ml 네임스페이스)로 fan-out합니다. seed/prompt/모델 버전을 기록해 재현성을 확보하고, 컷 계약과 캐릭터 identity를 추적해 일관성을 유지합니다. Higgs Field 기반 이미지·동영상 생성과 FLUX 계열 이미지 생성/수정 실험을 함께 사용해 컷 후보, 영상화 후보, 보정 후보를 자동 생성하는 흐름도 구성했습니다.

① GPU 호출 (렌더): ComfyUI를 풀 2종으로 운영합니다. comfyui-sdxl ×2(RTX 4070 12GB)는 SDXL(Illustrious / NoobAI / Animagine) + ControlNet(pose/depth/canny) 스테이지를, comfyui-large ×2(RTX 4060 Ti 16GB)는 Qwen-Image GGUF / Z-Image 등 VRAM이 큰 잡을 담당합니다. 클라이언트는 MURIM_COMFY_SDXL_ENDPOINTS / MURIM_COMFY_LARGE_ENDPOINTS로 pod별 엔드포인트 풀을 잡아 least-busy 선택 + 컷 단위 병렬로 호출합니다(단일 GPU 직렬 대비 wall-clock 약 1/4, 16GB 풀로 OOM 해소). 실제 렌더 그래프는 Qwen-Image-Edit 2511 fp8 + Lightning LoRA 기반으로, 확산 모델 / CLIP / VAE 로드 → TextEncodeQwenImageEditPlus → KSampler → VAE 디코드로 구성됩니다. 컷별 이미지 후보는 FLUX/ComfyUI 경로와 Higgs Field 생성 결과를 함께 비교하고, 영상 후보는 Higgs Field 기반 생성 결과를 검수 루프에 연결했습니다.

② LLM / VLM 호출: 같은 클러스터의 KubeRay + Ray Serve(OpenAI 호환, :8000/v1)에서 비전·텍스트 모델을 서빙합니다. qwen3-vl-30b-a3b(MoE)를 critic 주력 + 스토리보드 writer + reroll 프롬프트 수선에, internvl3-5-8b를 hero 컷 2차 교차검증에, qwen3-4b를 리포트/등록 payload에 사용합니다. STORY_MODEL=k8s로 라우팅하고 외부 codex CLI는 fallback으로 둡니다. critic은 에피소드 단위로 asyncio 동시 호출해 15컷 직렬 2–4min을 30s 미만으로 단축합니다.

③ 서버 인프라 / GPU 스케줄링: NVIDIA GPU Operator + HAMi DRA driver로 ComfyUI/Ray 워커가 총 16장 GPU 노드 풀(4060 Ti / 4070 / 4070 Ti)에서 GPU를 DRA claim 방식으로 공유하고, 유휴 워커는 scale-to-zero 됩니다. 모델 자산은 공유 /models/comfyui(Ceph) 볼륨에 두고 ml-model-download-* Job으로 체크포인트·ControlNet·LoRA를 주입하며, 클라이언트는 MURIM_MODEL_PROFILE=k8s 계약으로 동일 경로를 가정합니다. Mac↔클러스터는 VPN(utun100)으로 내부 API(10.31.10.10:6443)·ClusterIP(10.43/16)에 접속하고, 엣지는 Cilium Gateway가 담당합니다.

④ 파이프라인 흐름: 시놉시스 → 스토리보드(VLM) → renderability gate(결정적) → ComfyUI 풀 fan-out 렌더 → VLM critic(병렬, hero는 교차검증) → action router(결정적) → reroll 표적 수선(실패 이미지 입력) → Qwen image-edit repair → 자동 re-critic·등록·QA 리포트 → operator 최종 승인 → chocomics 플랫폼 발행. 사람은 시놉시스 작성·repair 승인·최종 승인만 맡고, 픽셀 개선은 자동 루프가 먼저 수행합니다.

HuriLLM

Groq, NVIDIA NIM, Google AI Studio, Cerebras, OpenRouter, Alibaba Model Studio처럼 무료 티어 또는 무료 크레딧을 제공하는 LLM API key를 한 곳에 등록하고, HuriLLM key 하나로 다시 호출할 수 있게 만든 통합 게이트웨이입니다.

제공사 key는 서버에 보관하고 사용자는 본인만의 HuriLLM API key를 발급·재발급해 사용합니다. 로컬 앱이나 내부 서비스에서는 base_url과 api_key만 HuriLLM 기준으로 바꿔 여러 제공사 모델을 같은 호출 흐름으로 사용할 수 있게 구성했습니다.

모델 카탈로그는 server/hosted-models.ts의 curated catalog와 제공사 /models 동기화 흐름을 합쳐 provider, builder, model id, context, parameter, category, recommended 여부를 보여줍니다. NVIDIA, Groq, Cerebras, OpenRouter, Alibaba, Google 모델을 한 화면에서 비교하고 사용 가능한 provider를 바로 확인할 수 있습니다.

API 키 화면은 HuriLLM key rotation과 제공사 key 등록·검증·삭제를 맡고, Playground는 등록된 provider key 기준으로 실제 모델 스트리밍 호출 또는 demo fallback을 수행합니다. Guide 화면에는 무료 키 발급 링크와 Python/cURL/Node.js OpenAI SDK 사용 예시를 함께 정리했습니다.

Playground에 웹 검색 토글을 붙인 이유는 무료·오픈 LLM이 최신 API 정책, 모델 릴리스, 문서 변경처럼 학습 시점 이후 정보에 약한 문제를 줄이기 위해서입니다. 검색을 켜면 Tavily로 최신 결과를 가져와 system context에 주입하고 출처를 함께 보여주기 때문에, 모델만 호출할 때보다 최신 정보 질문에서 환각 답변과 재질문이 줄고 작은 무료 모델도 근거 기반 답변을 만들 수 있게 개선했습니다. 기본값은 OFF로 두어 일반 대화의 지연 시간과 호출 비용은 그대로 유지하고, 최신성이 필요한 질문에서만 검색 오버헤드를 쓰도록 했습니다.

웹툰 학습 실패 경험과 방법 분석
failure log · training audit · rerender decision

웹툰 자동생성에서 가장 오래 잡고 있었던 부분은 “이미지가 나왔는가”가 아니라 “이 컷이 실제 에피소드에 들어갈 수 있는가”였습니다. 그래서 실패 이미지를 버리지 않고, 왜 실패했는지와 다음 실험에서 무엇을 바꿔야 하는지를 계속 기록했습니다.

학습과 렌더링을 반복하면서 얻은 결론은 단순했습니다. 더 오래 학습하거나 더 강한 모델을 쓰는 것보다, 데이터의 출처 다양성, 컨트롤 경로가 실제로 출력에 영향을 주는지, 부분 개선이 최종 strip 품질로 이어지는지를 먼저 증명해야 했습니다.

Problem
실패의 대부분은 “못 그린 이미지”가 아니라 “검증되지 않은 성공처럼 보이는 이미지”였습니다.

낮은 denoise의 img2img는 깨끗해 보였지만 사실상 원본 crop을 거의 보존한 결과였습니다. 이 방식은 현재 컷을 조금 정리하는 데는 쓸 수 있어도, 새 포즈와 새 표정에서도 캐릭터 identity가 유지되는지는 증명하지 못했습니다.

ControlNet도 노드가 들어갔다고 바로 믿지 않았습니다. SDXL 경로에서는 ControlNetApplyAdvanced가 workflow JSON에 존재했지만, 같은 seed의 no-control 출력과 pixel RMSE가 0.0000으로 같아 실제로는 제어가 먹지 않는 경로였습니다. 그래서 “워크플로우가 실행됨”과 “조건이 출력에 영향을 줌”을 분리해 보게 됐습니다.

Training
LoRA와 full fine-tune은 학습량보다 데이터 계약과 identity binding이 병목이었습니다.

초기 FLUX LoRA pack은 135장으로 보였지만 실제 source parent는 27개뿐이었습니다. 같은 원본에서 나온 crop variant를 여러 장 넣으면 학습 row 수는 늘지만, 새 장면에서 버틸 identity evidence는 늘지 않았습니다. 이후에는 strict-final 기준을 따로 두고, 최소 80개 이상의 고유 source parent와 source당 1장 원칙을 검사하도록 validation을 추가했습니다.

full checkpoint fine-tune도 같은 방식으로 검증했습니다. 360 step에서는 generic anime male로 drift했고, 720/1440 step까지 밀어도 얼굴이 “원본 캐릭터”가 아니라 sharp-eyed male bust로 수렴했습니다. 여기서 얻은 결론은 FLUX UNet을 더 오래 미는 것만으로는 부족하고, token/identity binding과 source-layout conditioning을 별도 문제로 풀어야 한다는 점이었습니다.

Rendering
부분 보정은 denoise, mask, reference를 작은 매트릭스로 쪼개서 판단했습니다.

고 denoise로 전체 얼굴을 다시 그리면 눈이 다시 열리거나 머리카락과 얼굴형이 드리프트됐습니다. Z-Image BF16 reference inpaint는 unmasked RMSE가 낮아 로컬 편집은 성공했지만, masked 영역이 checker/mosaic texture로 무너져 latent inpaint artifact로 분류했습니다.

Qwen crop edit은 crop-edit harness로는 유효했습니다. 다만 d0.45-d0.50은 원본 얼굴을 보존하는 데 가깝고, d0.52-d0.55부터 검은 눈가 밴드가 강해졌으며, d0.65+에서는 눈과 identity가 크게 바뀌었습니다. 그래서 “직접 품질 개선 route”로 승격하지 않고, crop/denoise 실험을 위한 제어된 도구로 남겼습니다.

검증 방식
same seed, RMSE/MAE, contact sheet, 최종 strip 흐름을 함께 봤습니다. 단일 PNG가 좋아 보여도 metadata와 비교 결과가 없으면 성공으로 보지 않았습니다.
데이터 관리
manual clean crop, source parent, caption, trigger token을 분리했습니다. row 수가 아니라 고유 source 수와 caption 다양성을 학습 승인 기준으로 삼았습니다.
개선 루프
실패 이미지는 폐기하지 않고 rerender 조건으로 되돌렸습니다. VLM critic, 손/소품 matrix, mask patch, multi-reference 비교를 다음 생성 조건에 반영했습니다.
Source-hugging 착시
낮은 denoise img2img는 선명하고 안정적으로 보였지만 대부분 원본 crop을 보존한 결과였습니다. 그래서 원본 보존형 repaint와 새 장면 생성 능력을 분리해 평가했습니다.
LoRA 회귀 분석
최신 학습본이 항상 더 좋지는 않았습니다. 좁은 얼굴 crop과 약한 caption 다양성은 몸 비율, 의상, 표정 안정성을 악화시켰고, 이후에는 학습 완료와 캐릭터 승인을 분리했습니다.
손·소품 오류 분해
cut05는 손가락 위치, 컵 중심선, 원근감이 조금만 어긋나도 장면 의도가 깨졌습니다. 그래서 denoise 0.08/0.16/0.24, 손 삽입 위치, 컵 보존 여부를 나눠 매트릭스로 만들고 실패 조건을 눈으로 검수했습니다.
마스크·레퍼런스 제어
FLUX 결과는 전체 품질이 높아도 얼굴/머리카락/손 영역이 드리프트될 수 있었습니다. mask patch와 multi-reference를 비교해, 전체 재생성보다 실패 영역만 제한적으로 보정하는 쪽이 에피소드 연속성을 더 잘 지킨다는 결론을 얻었습니다.
학습 인프라 실패도 분리
AdamW8bit OOM, gradient clipping stall, KubeRay worker 재생성, 모델 저장소 용량 부족처럼 품질과 무관한 실패도 따로 기록했습니다. 학습 profile 실패와 클러스터 preemption을 섞지 않기 위해 canary run과 guard를 둔 뒤 재실행했습니다.
최종 승인 기준
v024에서 v025처럼 일부 컷이 좋아져도 표정, 시선, 컵/손 관계, 컷 간 감정선이 약하면 승인하지 않았습니다. 후보는 항상 단일 컷이 아니라 세로 strip과 contact sheet 기준으로 판단했습니다.

Image / Video Models

ComfyUI GPU Pool

Scheduling / Infra
NVIDIA GPU Operator, HAMi DRA, KubeRay/Ray Serve, Ceph-backed /models/comfyui, Cilium Gateway 기반으로 모델 파일과 GPU 워커를 운영했습니다. ComfyUI/Ray 워커는 GPU claim과 scale-to-zero를 분리해 확인했습니다.
